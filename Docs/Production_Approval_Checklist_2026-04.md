# 운영 승인(Production Approval) 체크리스트 (2026-04)

이 문서는 현재 코드 상태에서 **운영 승인 보류**를 **승인 가능 상태**로 전환하기 위한 실무 기준입니다.

## 승인 기준(Exit Criteria)
아래 4개 게이트가 모두 통과되면 운영 승인 가능합니다.

1. **기능 게이트**: 핵심 명령(`/순위`, `/일정`) 정상 응답
2. **안정성 게이트**: 크롤링/DB 실패 시 봇 프로세스 생존 + 자동 복구
3. **운영 게이트**: `/health`로 장애 원인 즉시 파악 가능
4. **배포 게이트**: 동일 버전 재현 가능(의존성 고정 + 이미지 태깅)

---

## 1단계 (필수) — 코드 안전성 보강

### A. 스케줄러 명확화
- 현재 `@tasks.loop(hour=6)` 선언을 명확한 방식으로 교체
  - 예: `@tasks.loop(time=datetime.time(hour=6, tzinfo=datetime.timezone.utc))`
  - 또는 `@tasks.loop(hours=24)` + 최초 실행 시점 제어
- 완료 조건:
  - 부팅 후 스케줄 루프가 1회만 시작됨
  - 재연결 시 중복 루프 없음

### B. Selenium 자원 정리
- 모든 크롤링 함수에 `try/finally`로 `driver.quit()` 보장
- 완료 조건:
  - 실패/성공 모두 chrome 프로세스가 잔존하지 않음

### C. bare except 제거
- `except:`를 `except Exception as e:` 이상으로 변경하고 원인 로깅
- DB 오류, 파싱 오류, 네트워크 오류를 구분
- 완료 조건:
  - 로그만 보고 원인 분류 가능

---

## 2단계 (필수) — 운영 관측성/헬스체크

### D. `/health` slash command 도입
- 포함 항목:
  - DB 연결 성공/실패
  - 최근 크롤링 시각
  - 오늘 경기 건수
- 완료 조건:
  - 운영자가 디스코드 내에서 장애 상태를 30초 내 파악 가능

### E. healthcheck 실질화
- docker compose healthcheck를 단순 config 파싱이 아닌 앱 상태 기반으로 개선
- 완료 조건:
  - Discord 로그인 실패/DB 실패 시 unhealthy 전환

---

## 3단계 (필수) — 재현 가능한 배포

### F. 의존성 버전 고정
- `requirements.txt` 또는 `pyproject.toml` 도입
- Dockerfile은 해당 lock/requirements 기반 설치
- 완료 조건:
  - 팀원/서버에서 동일 버전으로 빌드 재현 가능

### G. 이미지/릴리즈 관리
- 이미지 태그 규칙: `app:<git-sha>`
- 롤백 규칙 문서화
- 완료 조건:
  - 장애 시 직전 안정 버전으로 10분 내 롤백 가능

---

## 4단계 (권장) — 승인 전 리허설

### H. 스테이징 3일 버너 테스트
- 시나리오:
  1. KBO 페이지 구조 변경 모의(파싱 실패)
  2. DB 일시 중단
  3. Discord API 연결 지연
- 완료 조건:
  - 봇 다운타임 없이 복구하거나 자동 재시작으로 회복

### I. 최소 테스트 자동화 강화
- 현재 AST 테스트 + 아래 추가:
  - DB 함수(mock) 테스트
  - 크롤러 파싱 함수 단위 테스트
  - `/health` 명령 테스트
- 완료 조건:
  - CI에서 핵심 테스트 green

## 운영 승인 회의에서 사용할 체크표

- [ ] 스케줄러 중복 실행 없음
- [ ] 크롤링 실패해도 프로세스 생존
- [ ] DB 장애 시 원인 로그 확인 가능
- [ ] `/health`로 상태 즉시 확인 가능
- [ ] 의존성 고정 + 재현 빌드 가능
- [ ] 롤백 절차 검증 완료
- [ ] 3일 스테이징 버너 테스트 통과

모든 항목이 체크되면 **운영 승인 권고** 가능합니다.
